# היחס בין ניהול תכנון למסלולים — Project Hub

**תאריך:** 2026-06-21
**קובץ יעד:** `C:\Users\Omega\Database\project_hub.html`

## רקע

ב-Project Hub יש שתי תצוגות שמציגות חלק חופף של אותו מידע:
- **מסלול רישוי** — sub-tabs, אחד per שלב רישוי (sheet מסוג `list`), כל שלב עם ה-`ListView` המלא שלו.
- **ניהול תכנון** (`NihulTikhnunView`) — מציג כיום את כל שלבי הרישוי ברצף אחד, כל שלב עם ה-`ListView` שלו (ראו commit `0882d9f`).

בנוסף קיים sheet ריק ולא-משומש בשם "ניהול תכנון" שנוצר עם הפרויקט (`makeEmptySheet('ניהול תכנון')`, ב-`makeDefaultData()`), שמוחרג כיום מהרשימה לפי שם.

## בעיה

1. אין דרך לתאר משימה שמקדמת תכנון (מתכנון ראשוני ועד תכנון מפורט) בלי שתהיה שייכת לשלב רישוי קונקרטי. כל משימה חייבת לחיות בתוך `sheet.tasks` של שלב מסוים.
2. אין הפרדה בין "משימות רישמיות של שלב" לבין "משימות רישוי כלליות" (לא קשורות לשלב ספציפי) ו"משימות שוטפות" (לא קשורות לרישוי בכלל).
3. ניהול תכנון צריך להיות ה-source of truth שמכיל את כל אלה, בעוד שמסלול רישוי צריך להישאר תצוגה ממוקדת רק לשלבים הרישמיים שלו.

## החלטה

**לא מוסיפים שדה `track` למשימה.** ההפרדה בין הקטגוריות נעשית ברמת ה-**sheet**, לא ברמת המשימה:

| קטגוריה | ייצוג מבני |
|---|---|
| משימה שנגזרת מדרישה של משימת-שלב | תת-משימה (subtask) של משימת השלב — **קיים היום, ללא שינוי** |
| משימות רישוי כלליות (לא משולבות-שלב) | sheet רגיל נוסף בתוך מסלול רישוי, למשל "כללי - רישוי" — נוצר ע"י המשתמש בדיוק כמו כל שלב, **אין UI מיוחד נדרש** |
| משימות שוטפות (לא קשורות לרישוי בכלל) | ה-sheet הריק הקיים "ניהול תכנון" — **משנים שם ל"משימות שוטפות"** והופך לשמיש |

כל sheet מקבל שדה חדש `track`, שמשמש רק כדי לדעת **איפה הוא מוצג**, לא כדי לשנות איך הוא מתנהג:

```js
sheet: {
  ..., // ללא שינוי במבנה הקיים
  track: 'licensing' | null,   // חדש. null = לא שייך לשום מסלול ("שוטף")
}
```

## מיגרציה (`migrateData`)

שלב מיגרציה חדש, רץ אחרי שלבי המיגרציה הקיימים:

1. לכל sheet מסוג `'list'`:
   - אם `sheet.name === 'ניהול תכנון'` (ה-placeholder הריק הקיים) → `name: 'משימות שוטפות'`, `track: null`. שמירת ה-`id` הקיים (לא יוצרים sheet חדש).
   - אחרת → `track: sheet.track ?? 'licensing'` (כל שלב קיים הופך אוטומטית לרישוי, ברירת המחדל ההיסטורית).
2. אידמפוטנטי: בדיקת `'track' in sheet` לפני שמריצים, כמו בתבנית המיגרציה הקיימת ל-`events` (ראו `2026-06-16-unified-process-model-design.md`).
3. sheet חדש שנוצר מתוך מסלול רישוי (`addSheet()` כשנקראת מתוך track==='licensing') מקבל `track:'licensing'` אוטומטית.

## סינון לפי track

**מסלול רישוי** (sub-tabs ו-`DashboardView`):
```js
sheets.filter(s => s.type==='list' && s.track==='licensing')
```
מחליף את הסינון הנוכחי לפי שם (`s.name!=='ניהול תכנון'`) בכל מקום שבו הוא מופיע (sub-tabs, `DashboardView` cards/gantt, `addSheet`).

**ניהול תכנון** (`NihulTikhnunView`):
מקבל **שורת לשוניות-משנה** בראש הדף (לא קיבוץ של מדורים מתחת זה לזה ברשימה אחת — יותר מדי מידע בבת אחת):
- לשונית "ניהול שוטף" — מציגה רק את ה-sheet עם `track===null` (כיום sheet יחיד, "משימות שוטפות").
- לשונית "מסלול רישוי" — מציגה את כל ה-sheets עם `track==='licensing'`, ערוכים זה תחת זה בדיוק כמו המימוש הקיים היום (`ListView` per sheet, ראו commit `0882d9f`) — רק בלי שמשימות השוטף מוצגות יחד איתם.
- לשונית פעילה כברירת מחדל: "ניהול שוטף".
- מבנה הלשוניות מתרחב באופן ישיר להוספת לשונית per track נוסף בעתיד (מכרז/ביצוע) — אותה רשימת tracks שמזינה את הסינון מזינה גם את רשימת הלשוניות.

## הרחבה עתידית (מחוץ לסקופ, לא נחסם)

כשמכרז/ביצוע יקבלו sheets משלהם עם `track:'tender'|'execution'`, הם:
- יקבלו לשונית-משנה נוספת בניהול תכנון אוטומטית, בלי שינוי קוד נוסף במנגנון הלשוניות (אותה רשימת tracks).
- יקבלו טאב/תצוגה ראשי משלהם בסיידבר רק כשתיבנה הלוגיקה הפרטנית שלהם (לא חלק מה-spec הזה).

הלוגיקה הפנימית של מכרז וביצוע (מעקב מסמכים ממתכננים, יחס מול קבלן) **אינה** חלק מהמסמך הזה.

## טיפול בקצוות

- **sheet "כללי - רישוי" לא קיים עדיין בדאטה הקיים** — לא נוצר אוטומטית במיגרציה; המשתמש יוצר אותו ידנית כשירצה (כמו כל שלב), ע"י כפתור "+" הקיים בתוך מסלול רישוי.
- **"משימות שוטפות" מתחיל ריק** — בדיוק כמו היום, רק ניתן להזין בו תוכן.
- **sheets מסוג `'dashboard'`** — לא מושפעים מ-`track`; הם מוצגים רק בתוך `licensing` כפי שהם מוצגים היום (אין dashboard כללי בשלב הזה).
- **דאטה ישן ב-localStorage (`pm_asana_v9`)** — מטופל ע"י המיגרציה; לא נדרש bump של `STORAGE_KEY`.

## בדיקות (טסטים ידניים)

1. טעינת דאטה ישן → כל שלבי הרישוי הקיימים מקבלים `track:'licensing'`, ה-sheet הריק הופך ל"משימות שוטפות" עם `track:null`, בלי לאבד אף משימה קיימת.
2. מסלול רישוי מציג רק שלבי `track==='licensing'` — "משימות שוטפות" לא מופיע בו כ-sub-tab.
3. ניהול תכנון מציג שתי לשוניות-משנה: "ניהול שוטף" (פעילה כברירת מחדל, עם ה-sheet שהיה ריק) ו"מסלול רישוי" (עם כל השלבים ערוכים זה תחת זה). מעבר בין הלשוניות לא מציג את שתי הקבוצות יחד.
4. יצירת sheet חדש "כללי - רישוי" מתוך מסלול רישוי → מקבל `track:'licensing'` אוטומטית, מופיע גם כ-sub-tab במסלול רישוי וגם בלשונית "מסלול רישוי" בניהול תכנון.
5. הוספת משימה ל"משימות שוטפות" מתוך ניהול תכנון נשמרת נכון (`saveSheet` הקיים ב-`NihulTikhnunView`) ולא מופיעה בטעות במסלול רישוי.
6. הרצת `migrateData` פעמיים על אותו דאטה → לא משנה תוצאה (אידמפוטנטי).
