# איחוד מודל המשימות (simple/process) — Project Hub

**תאריך:** 2026-06-16
**קובץ יעד:** `C:\Users\Omega\Database\project_hub.html`

## רקע

כיום ב-Project Hub יש הפרדה מפורשת בין שני סוגי משימות:
- `taskType: 'simple'` — משימה רגילה, done/not-done בלבד
- `taskType: 'process'` — משימה עם טיימליין של events (progress/sent/response/comments/approved)

המשתמש בוחר את הסוג מפורשות (לחיצה ימנית על כפתור הסטטוס מחליפה ביניהם). אותה הפרדה קיימת גם בתת-משימות, עם דגל `isProcess`.

## בעיה

ההפרדה המפורשת מכריחה את המשתמש להחליט מראש "האם זו תהיה משימת תהליך" — לפני שיש בכלל עדכון לתאר. זה לא תואם לשימוש בפועל: לרוב לא ידוע מראש אם משימה תצטרך טיימליין עדכונים, וזה מתברר רק כשמגיע עדכון ראשון.

## החלטה

מאחדים את המודל: **אין יותר שדה נבחר-מראש שמגדיר את סוג המשימה.** כל משימה (וכל תת-משימה) יכולה לקבל עדכונים בטיימליין בכל רגע. הסטטוס הוויזואלי (האם "נראית" כתהליך) **מחושב** מתוך מספר ה-events הקיימים, לא נשמר כדגל נפרד.

זה חל **גם על משימות ראשיות וגם על תת-משימות** — איחוד מלא, כולל הסרת `isProcess` מתת-משימות.

## מודל נתונים

### לפני
```js
task: {
  ..., taskType: 'simple'|'process',
  processEvents?: [...],          // legacy, ראו migrateData
  subtasks: [{
    ..., isProcess?: boolean,
    events?: [...]
  }]
}
```

### אחרי
```js
task: {
  ...,                              // אין taskType
  events: [...],                    // תמיד מערך (יכול להיות ריק), לא אופציונלי
  subtasks: [{
    ...,                            // אין isProcess
    events: [...]                   // תמיד מערך (יכול להיות ריק)
  }]
}
```

**כלל הצגה:** `hasProcess(entity) = entity.events.length > 0`. אין שדה בוליאני נשמר — זה תמיד נגזר.

### מיגרציה (`migrateData`)

נדרש שלב מיגרציה חדש שרץ אחרי `processToSubtask`/`hoistEvents` הקיימים:
1. לכל task: לאחד legacy `processEvents` (אם יש) לתוך `task.events`. למחוק `taskType` ו-`processEvents` מהאובייקט.
2. לכל subtask: להעביר `events` הקיים כמו שהוא (כבר באותו שם). למחוק `isProcess`.
3. אם ל-task/subtask אין `events` בכלל — להוסיף `events: []`.
4. המיגרציה היא **one-way** ו-**idempotent**: הרצה חזרה על דאטה שכבר עברה מיגרציה לא משנה כלום (בודקים `'taskType' in task` לפני שמריצים).

לא נדרשת תאימות לאחור לקריאה בפורמט הישן — `loadData()` תמיד מריץ מיגרציה לפני שהדאטה מגיעה ל-state.

## עיצוב ויזואלי

### אייקון הסטטוס (כפתור בתחילת השורה)
- `events.length === 0` → `⭕` (כפתור `done-btn` הקיים, לחיצה = toggle done)
- `events.length > 0` → `◐` (כפתור `proc-btn` הקיים, **אותו גודל** כמו `done-btn` — לא לשנות מידות/padding)
- האייקון מחושב בכל רינדור מ-`events.length`, לא נשמר. מחיקת כל ה-events מחזירה אוטומטית ל-`⭕`.
- מוסר ה-context-menu (קליק ימני) שהיה מחליף `taskType` — אין יותר "המרה" מפורשת.

### תג מספר עדכונים
- כש-`events.length > 0`, מוצג תג קטן לצד הכותרת: `{events.length} עדכונים` — עיצוב עקבי עם `sub-badge` הקיים (`className="sub-badge"` או קלאס דומה ייעודי).
- התג נעלם אוטומטית כש-`events.length === 0`.

### כפתור "+ עדכון"
- מופיע **רק ב-hover** מעל שורת המשימה (או תת-המשימה) — לא קבוע, לא תופס מקום כשהעכבר לא מעל השורה.
- ממוקם לצד כפתור הסטטוס/הכותרת.
- לחיצה פותחת את אותו ממשק הוספת event שקיים היום לטיימליין תהליך (טופס/inline להוספת event עם status/assignee/date/note).
- זמין תמיד — בלי קשר אם יש כבר events או לא. זו נקודת הכניסה היחידה ליצירת event ראשון.
- חל גם על תת-משימות, באותה לוגיקה.

## טיפול בקצוות

- **משימה עם 0 subtasks ו-0 events:** מוצגת בדיוק כמו "simple" היום — אין שינוי התנהגות.
- **מחיקת event בודד מתוך כמה:** אם נשאר `events.length>0` — נשאר ◐ והתג מתעדכן למספר הנכון. אם זה היה האחרון — חוזר ל-⭕ והתג נעלם.
- **נתונים ישנים בלוקאלסטורג' (`pm_asana_v9`)** שעדיין כוללים `taskType`/`isProcess`: מטופלים ע"י המיגרציה החדשה בטעינה; לא נדרש bump של STORAGE_KEY כי המיגרציה היא תוספתית על אותו מבנה.
- **ביצועים:** `events.length` הוא O(1) על מערך קיים בזיכרון — אין השפעת ביצועים מורגשת.

## בדיקות (טסטים ידניים, אין מסגרת אוטומטית בקובץ HTML בודד)

1. משימה חדשה נוצרת → מוצגת ⭕, ללא badge, "+ עדכון" מופיע רק ב-hover.
2. הוספת event ראשון → האייקון מתעדכן ל-◐, badge מציג "1 עדכונים", בלי לרענן ידנית.
3. הוספת עוד events → המספר בתג עולה.
4. מחיקת כל ה-events → חזרה ל-⭕ ולהיעלמות התג.
5. אותו תהליך על subtask.
6. טעינת דאטה ישן (עם `taskType`/`isProcess` שמורים מראש בלוקאלסטורג') → המיגרציה רצה בלי שגיאות, ההתנהגות תואמת את הכלל החדש.
7. הרצת `migrateData` פעמיים על אותו דאטה (simulate reload) → לא משנה תוצאה (idempotent).
